قلب المشروع — بيتحكم في كل الطلبات

إيه دوره؟
ده الـ API server — يعني لما الـ browser يبعت طلب (upload CV، get jobs، apply)، main.py هو اللي بيستقبله ويرد عليه. بيستخدم FastAPI وهو framework بيخلي إنشاء API سهل وسريع.
  • 1
    Endpoints: كل وظيفة ليها URL خاص بيها. مثلاً /candidates/upload لرفع CV و /jobs لجلب الوظائف.
  • 2
    Lifespan: لما السيرفر يشتغل أول مرة، بيعمل init_db() عشان يتأكد إن الـ tables موجودة في الـ database.
  • 3
    Dependencies: بيستخدم Depends(get_db) — دي طريقة FastAPI عشان يديك connection للـ database في كل endpoint تلقائياً.
  • 4
    CORS Middleware: بيسمح لأي موقع يكلم الـ API — مهم عشان الـ frontend والـ backend ممكن يكونوا على domains مختلفة.
# مثال: endpoint بيرفع CV @app.post("/candidates/upload") async def upload_cv(file: UploadFile): content = await file.read() # اقرأ الملف parsed = parse_cv(content) # استخرج البيانات db.add(candidate) # احفظ في DB return candidate # رد على الـ browser

database.py

تعريف الـ tables وإزاي البيانات بتتحفظ

إيه دوره؟
بيعرّف شكل الـ database كاملاً. بيستخدم SQLAlchemy ORM — يعني بدل ما تكتب SQL بتكتب Python وهو بيترجمها لـ SQL تلقائياً.
  • 1
    Candidate: table بتحفظ بيانات كل مرشح — الاسم، الإيميل، سنوات الخبرة، التعليم، الـ skills.
  • 2
    JobDescription: table بتحفظ الوظائف — العنوان، الوصف، الخبرة المطلوبة، واتساب وإيميل التواصل.
  • 3
    Skill: table مشتركة بين المرشحين والوظائف — علاقة Many-to-Many يعني مرشح واحد ممكن يبقى عنده أكتر من skill ووظيفة واحدة ممكن تطلب أكتر من skill.
  • 4
    Application: table بتحفظ طلبات التوظيف — مين apply لأنهي وظيفة وإيه الـ score.
  • 5
    init_db(): بتعمل الـ tables لو مش موجودة، وبتعمل ALTER TABLE بأمان لو في columns جديدة على قاعدة بيانات قديمة.
# العلاقة بين المرشحين والـ skills (Many-to-Many) candidate_skills = Table( "candidate_skills", Column("candidate_id", ForeignKey("candidates.id")), Column("skill_id", ForeignKey("skills.id")), )

matcher.py

خوارزمية حساب نسبة التطابق

إيه دوره؟
ده أهم ملف في المشروع — بيحسب نسبة تطابق المرشح مع الوظيفة (من 0 لـ 100). النسبة بتتحسب من 3 عوامل بأوزان مختلفة.
Score = (Skills × 55%) + (Experience × 30%) + (Education × 15%)
  • 1
    Skill Score (55%): كم skill من المطلوب موجودة عند المرشح؟ لو عنده 4 من 5 skills = 80%.
  • 2
    Experience Score (30%): لو الوظيفة تطلب 3 سنين والمرشح عنده 5 = 100%. لو عنده 1.5 سنة = 45%.
  • 3
    Education Score (15%): بيرتب الدرجات: High School → Diploma → Bachelor → Master → PhD. لو المرشح أعلى أو بالظبط = 100%.
  • 4
    Semantic Boost (bonus): لو عندك OpenAI API key، بيستخدم AI embedding عشان يشوف Skills متشابهة في المعنى مش بس في الكلمة (مثلاً "JS" و "JavaScript").
# مثال: حساب skill score candidate = {"python", "sql", "django"} required = {"python", "sql", "react", "docker"} matched = candidate & required # → {python, sql} score = 2 / 4 * 100 # → 50%
Weighted Scoring Set Intersection Embeddings (optional)

ai_service.py

التواصل مع الـ AI (Groq / LLaMA)

إيه دوره؟
كل الـ AI features بتمشي من هنا. بيبعت prompts لـ Groq API (LLaMA model) ولو فيه مشكلة بيعمل fallback بـ logic عادي من غير AI.
  • 1
    parse_cv_claude(): بياخد نص الـ CV الخام ويبعته للـ AI عشان يستخرج منه الاسم، الإيميل، الـ skills، سنوات الخبرة — بيرجع JSON منظم.
  • 2
    career_coach_analysis(): بيعمل تقرير كامل للمرشح — نقاط القوة، الـ skill gaps، خطة عمل، career roadmap.
  • 3
    cv_score_analysis(): بيدي الـ CV درجة من 100 مع تفاصيل كل section (Contact, Experience, Skills, Education, Structure).
  • 4
    interview_prep(): بيعمل 8 أسئلة مقابلة مخصصة للوظيفة — 3 technical + 3 behavioral + 2 situational.
  • 5
    salary_estimate(): بيقدر الراتب في الإمارات بالدرهم بناءً على الـ skills والخبرة والتعليم مع مقارنة بـ benchmarks حقيقية.
  • 6
    Fallback: لو Groq ما ردش أو فيه error، كل function عندها نسخة بـ Python عادية تقدر تشتغل من غير AI.
# الـ pattern المتكرر في كل function def any_feature(data): raw = _generate(prompt) # بعت للـ AI if raw: return json.loads(raw) # رد AI return _fallback(data) # fallback

استخراج البيانات من ملفات PDF و DOCX

إيه دوره؟
بياخد ملف CV (PDF أو DOCX) ويحوله لـ بيانات منظمة. خطوتين: أول يستخرج النص الخام من الملف، بعدين يحلله ويجيب المعلومات المهمة.
  • 1
    PDF extraction: بيستخدم مكتبة pdfminer أو pypdf2 عشان يستخرج النص من الـ PDF.
  • 2
    DOCX extraction: بيستخدم مكتبة python-docx عشان يقرأ ملفات Word.
  • 3
    AI Parsing: بيبعت النص لـ ai_service.parse_cv_claude() عشان يستخرج البيانات بشكل ذكي.
  • 4
    KNOWN_SKILLS list: قائمة بأكتر من 200 skill تقني ومهني — بتستخدمها matcher.py عشان يعرف يحلل الـ job descriptions ويجيب منها الـ skills.
# الخطوات بالترتيب def parse_cv(content, filename): # 1. استخرج النص الخام text = extract_text_from_pdf(content) # 2. بعت للـ AI parsed = parse_cv_claude(text) # 3. لو AI فشل → regex fallback return parsed or regex_parse(text)

مفاهيم مهمة ممكن يسألوك عنها

ليه استخدمنا FastAPI مش Flask؟

سؤال ممكن يتسألك في الـ interview

الإجابة:
FastAPI بيدعم async/await يعني ممكن يستقبل ملفات كبيرة ويبعتها للـ AI في نفس الوقت من غير ما يـblock الـ server. كمان بيعمل API documentation تلقائي على /docs، وعنده type validation تلقائي مع Pydantic. Flask أبسط لكن FastAPI أسرع وأكثر type-safe.

إيه هو الـ ORM وليه استخدمناه؟

سؤال ممكن يتسألك في الـ interview

الإجابة:
ORM = Object Relational Mapper. بدل ما تكتب SQL بتاعت: SELECT * FROM candidates WHERE email = 'x' بتكتب Python عادي: db.query(Candidate).filter(Candidate.email == 'x') الفايدة: أكثر أماناً من SQL injection، سهل تغير الـ database من PostgreSQL لـ MySQL من غير ما تغير الكود.

إيه هو الـ Fallback وليه مهم؟

سؤال ممكن يتسألك في الـ interview

الإجابة:
الـ Fallback هو الخطة البديلة لو الـ AI مش شغال. مثلاً لو Groq API وقع أو خلص الـ credits، المشروع مش بيوقع — بيشتغل بـ regex وقواعد بسيطة.

ده مبدأ مهم في الـ software engineering اسمه Graceful Degradation — يعني لو جزء وقع، باقي الموقع يشتغل بكفاءة أقل مش يوقع خالص.
